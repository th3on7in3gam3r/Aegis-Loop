    How to use Overview

    Your security command center for code

    What is this? Overview is the home screen of the Code module. It shows how healthy your repositories are at a glance — how many critical issues you have, your average security score, which repos need attention, and how much you have already fixed. Before your first scan you will see a getting-started hero; after scanning, KPI cards and charts replace it.

    Why it matters: You do not need to open every repo to know where risk lives. Overview rolls up the latest scan per repository so one number tells you if you are improving or slipping. Use it in standups, before releases, or when onboarding a new engineer.

    When to use it: Start here every session. Scan from + New scan or Repositories, then return to Overview to see the workspace picture change. Drill into Findings for the full list or A-Fix queue to remediate.

    Overview (big picture) → Findings (every issue) → A-Fix (patch) → PR (ship)
    1. 1
      Run your first scanClick Run demo scan on the hero below, or + New scan in the header for your own repo. Until you scan, charts stay empty — that is normal.
    2. 2
      Read the KPI stripAfter scanning, five cards appear: Critical, High, Fixed, Security score, and Repos scanned. The label Workspace · latest scan per repo means we dedupe — one scan per repo, not every historical run.
    3. 3
      Use the chartsFindings by severity — bar chart of open issues. Security score — ring shows workspace average (higher is better). Repository coverage — which repos have the most open findings. Remediation progress — fixed vs still open.
    4. 4
      Take actionOpen Findings in the sidebar to filter and click A-Fix on any row. The sidebar badges on Findings and A-Fix queue update from the same data as Overview.

    What the numbers mean

    • Critical / High — open findings by severity from latest scans (secrets and SQLi are usually Critical).
    • Fixed — findings you resolved with A-Fix or marked fixed in a scan.
    • Security score — 0–100 per repo; workspace card shows the average. Drops when critical/high issues pile up.
    • Repos scanned — count of repositories with at least one completed scan here.
    • Findings badge — same open count as the severity chart total.

    Where to go next

    • Findings — full searchable table of every open issue across repos.
    • A-Fix queue — findings that have a template or AI fix ready.
    • Repositories — scan any GitHub repo you can access.
    • Cloud / Attack / Protect — switch modules in the top pill bar for infra, live URLs, and WAF.
    Why is Critical 0 but the chart shows issues?

    Severity labels differ: KPI High maps to scanner “warning” severity (e.g. vulnerable dependencies). Check the chart legend for the critical / high / info split.

    Why did my score disappear after idle time?

    Free workspaces sleep when unused. Scan history may reset — reconnect GitHub and rescan. Your Overview repopulates from new scans.

    Is this mock data?

    No. Every number comes from real scans you ran (demo or your repos). The demo uses a sample app with intentional vulnerabilities.

    Overview vs Findings — which should I use?

    Overview for trends and “are we getting better?” Findings when you need file paths, rule IDs, and A-Fix buttons for specific issues.

    How often should I scan?

    On every meaningful PR, weekly on main branches, and before releases. Re-scan after applying A-Fix to see Fixed and Remediation progress move.

    Can I see one repo only?

    Overview is workspace-wide. Click a repo in Recent scans or open Findings and filter — or run a single-repo scan and check that scan’s KPI scope (“This scan”) in the strip.

    How to use Cloud

    Catch cloud misconfigs before they go live

    What is this? Cloud scans the infrastructure files in your repo — Terraform, Kubernetes YAML, Docker Compose — for settings that accidentally expose data to the internet (public storage buckets, firewalls open to everyone, etc.). You do not need AWS or GCP credentials; we read the files, not your live cloud account.

    Why it matters: A single misconfigured S3 bucket or security group can leak customer data even when your application code is perfect. This module finds those mistakes in the same place you already edit code.

    When to use it: Before merging infra changes, after onboarding a new repo with Terraform/K8s, or when auditing “why is this port public?” Run Cloud alongside Code scans — app bugs and infra gaps are different problems.

    Code finds app bugs → Cloud finds infra exposure → Protect blocks live attacks
    1. 1
      Try the demo scanClick Run demo scan below. You will see sample findings from a bundled Terraform + Kubernetes + Docker repo — no GitHub connection required. Use this to learn what a “Critical” cloud finding looks like before scanning your own stuff.
    2. 2
      Scan your own repoClick Scan repository IaC and enter owner/repo (e.g. acme-corp/platform) plus branch main. We clone the repo and scan every .tf, .yaml, and Docker file we find — same GitHub access as the Code module.
    3. 3
      Fix in your IaCEach row shows file, line, and rule ID. Open that file in your editor: replace 0.0.0.0/0 with your office/VPN CIDR, set S3 block_public_acls = true, narrow IAM Action lists, or add a NetworkPolicy in front of a public LoadBalancer.
    4. 4
      Feed ProtectSwitch to the Protect tab → Sync rules from findings. Cloud findings like open security groups can add SSRF-blocking rules so metadata IP probes are stopped at the edge.

    What we check (by provider)

    AWS

    Public S3 ACLs, security groups open to 0.0.0.0/0, wildcard IAM Action: "*".

    Kubernetes

    Services with type: LoadBalancer that expose cluster workloads publicly.

    Docker / GCP / Azure

    Ports on 0.0.0.0, GCS allUsers ACLs, Azure NSGs with source_address_prefix = "*".

    Reading your results

    • Critical — likely internet-wide exposure (public bucket, world-open firewall). Fix before deploy.
    • High — risky defaults (wildcard IAM, public LoadBalancer). Confirm intent or restrict.
    • Posture score — higher is better; drops as critical/high findings accumulate.
    • Rule column — stable ID like cloud/sg-open-world for tickets and Protect sync.

    Example workflow

    • Developer opens PR changing infra/main.tf → run Cloud scan on that repo.
    • Finding: Security group open to the world on line 42 → change CIDR to 10.0.0.0/8.
    • Re-scan until Critical = 0 → merge PR → sync Protect for runtime SSRF rules.
    What files does Cloud scan?

    Files ending in .tf, .yaml, .yml, .json, .hcl, plus paths named docker-compose, cloudformation, Dockerfile, or serverless. We pattern-match dangerous strings — we do not execute Terraform or connect to your cloud API.

    Do I need to connect AWS, GCP, or Azure?

    No. Cloud analyzes what is declared in git. That catches the misconfig before it is applied. Live cloud inventory scans (comparing running resources vs. code) are on the roadmap.

    Why did my scan find nothing?

    Common reasons: the repo has no IaC files, infra lives in a different repo you did not scan, or your templates already follow least-privilege. Try the demo scan first to confirm the module is working.

    Is a LoadBalancer finding always wrong?

    Not always — public LBs are intentional for many APIs. Treat High findings as “confirm this is meant to be public, then add network policies or WAF rules.”

    How is this different from Code scanning?

    Code finds vulnerabilities in application source (SQLi, secrets in JS). Cloud finds exposure in infrastructure definitions — the layer that decides who on the internet can reach your storage and ports.

    Can I scan the same repo on every PR?

    Yes. Re-run Scan repository IaC whenever infra files change. Pair with Code PR scans so both app and infra are checked before merge.

      How to use Attack

      Test what attackers see on your live site

      What is this? Attack sends a safe, read-only request to a URL you choose (your staging site, production app, landing page) and checks basic security hygiene: is HTTPS enforced? Are protective headers present? Does the server leak too much info?

      Why it matters: Code scans catch bugs in source. Attack checks what is actually exposed on the web right now — the same surface real attackers probe. A perfect codebase behind a misconfigured CDN can still be vulnerable.

      When to use it: After deploying to staging or production, when onboarding a new domain, or before a launch checklist. Probe staging first, then production once you are confident.

      Deploy site → Attack probes headers & HTTPS → fix CDN/server config → Protect blocks payloads
      1. 1
        Pick a URL you controlUse a full URL like https://staging.yourapp.com or just yourapp.com (we add https:// automatically). Only probe sites you own or have written permission to test — never third-party sites.
      2. 2
        Run a probeClick Probe URL, paste the address, submit. We send a single GET request with redirects followed — no POST fuzzing, no login attempts, no exploit payloads.
      3. 3
        Fix what we findMissing HSTS? Add Strict-Transport-Security at your CDN or nginx. Missing CSP? Start with a report-only policy. Plain HTTP? Force redirect to HTTPS. Server header too verbose? Strip or genericize it.
      4. 4
        Sync to ProtectOn the Protect tab, sync rules from findings. For example, sites still on HTTP can add rules that block cleartext URL patterns in incoming requests.

      What each check means

      HTTPS / HSTS

      Critical if the site answers on plain HTTP. HSTS tells browsers to always use TLS — prevents sslstrip-style downgrade attacks.

      CSP & X-Frame-Options

      Content-Security-Policy limits where scripts can load. X-Frame-Options reduces clickjacking (your page embedded in a malicious iframe).

      Server disclosure

      Info-level finding when the Server header reveals nginx/Apache/IIS versions — helps attackers pick exploits.

      Reading your results

      • Critical — site served over unencrypted HTTP (attack/plain-http).
      • High — missing HSTS or CSP (attack/missing-hsts, attack/missing-csp).
      • Info — missing X-Frame-Options, nosniff, or verbose Server header.
      • Surface score — summary metric; fewer header/HTTPS gaps = higher score.
      • Probe failed? — timeout, DNS error, or blocked bot traffic. Check the URL is reachable from the public internet.

      Example fixes (nginx)

      • add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
      • add_header Content-Security-Policy "default-src 'self';" always;
      • add_header X-Frame-Options "SAMEORIGIN" always;
      • Redirect HTTP → HTTPS with return 301 https://$host$request_uri;

      Cloudflare, Vercel, and AWS CloudFront have equivalent settings in their dashboards — you do not need nginx specifically.

      Is this a full penetration test?

      No. Attack runs passive surface checks: one GET, response headers, status code. It does not brute-force passwords, scan for SQLi, or attempt to exploit CVEs. Hire a professional pentest for that scope.

      What URLs can I probe?

      Any URL you are authorized to test — your apps, staging environments, marketing sites, and customer deployments you manage. Avoid third-party sites, competitor domains, and government systems without explicit approval.

      Will probing affect my production traffic?

      Each probe is a single GET request — negligible load. Still, start with staging. Some WAFs may rate-limit our scanner user-agent (AegisLoop-Attack/1.0); whitelist if needed.

      Why does a “secure” site still show findings?

      Headers are often set at the CDN edge but missing on origin, or vice versa. We report what the URL returns after redirects — fix at whichever layer serves the final response.

      Attack vs. Code — which first?

      Code first (find bugs in source), then Attack on deployed URLs (verify headers/HTTPS in production), then Protect (block automated probes). They complement each other; neither replaces the other.

      Can I probe localhost?

      Our server runs in the cloud and cannot reach localhost on your laptop. Use a public staging URL or a tunnel (ngrok, Cloudflare Tunnel) for local testing.

        How to use Protect

        Block attacks at the edge — automatically

        What is this? Protect is a web application firewall (WAF). It watches incoming HTTP requests and blocks obvious attack patterns — SQL injection strings, script tags in URLs, path traversal (../) — before they reach your application code.

        Why it matters: Even with clean code and solid cloud posture, automated bots constantly scan the internet for SQLi and XSS probes. Protect is the last line when something slips through review or a zero-day appears.

        When to use it: After your first Code/Cloud/Attack scans (so derived rules exist), before exposing an app publicly, and whenever you see suspicious traffic in logs. Think of it as a safety net, not a substitute for fixing root causes.

        Scan (Code / Cloud / Attack) → Sync rules → toggle as needed → Block log confirms blocks
        1. 1
          Run other modules firstScan repos in Code (secrets, SQLi), IaC in Cloud (open SGs), URLs in Attack (HTTP downgrade). Protect maps specific finding types to extra WAF rules — more scans = smarter coverage.
        2. 2
          Sync rulesClick Sync rules from findings. You always get 3 built-in rules (SQLi, XSS, path traversal) plus optional derived rules from your findings. Re-sync after new scans to pick up fresh mappings.
        3. 3
          Enable or disable rulesEach row has Enable/Disable. Turn off a rule if it blocks legitimate traffic (rare, but possible with aggressive patterns). The Blocks column shows how often each rule has fired.
        4. 4
          Test a blockClick Test block (SQLi) — we send a harmless ' OR 1=1 -- payload to the API. If blocked, you will see a toast and a new row in the Block log below.

        On this dashboard, Protect is live — malicious query strings sent to /app or /api are blocked here so you can see it working. In production, deploy the same rule patterns in front of your app (Cloudflare WAF, AWS WAF, nginx, or middleware).

        Built-in rules (always present)

        SQL injection

        Blocks patterns like UNION SELECT and OR 1=1 in query strings and request bodies.

        XSS

        Blocks <script tags and javascript: URLs in incoming requests.

        Path traversal

        Blocks ../ sequences that attackers use to read files outside the web root.

        Derived rules (from your scans)

        • Code — AWS key patterns in requests, SQLi payloads from injection findings.
        • Cloud — SSRF probes to cloud metadata IPs (169.254.169.254) when open SG findings exist.
        • Attack — cleartext HTTP URL patterns when a site is still served over HTTP.

        Reading the dashboard

        • Rules — total rule count (built-in + derived).
        • Active — rules currently enabled and enforcing.
        • Blocked — cumulative blocks across all rules since last reset.
        • Block log — last 20 blocked requests with method, path, matched rule, and timestamp.
        Where do rules come from?

        Three sources: (1) built-in patterns always loaded on sync, (2) rules mapped from Code/Cloud/Attack finding rule IDs, (3) your enable/disable choices per rule. Sync merges new findings without duplicating existing rule IDs.

        Will this break my app?

        Rules only block requests matching attack patterns — normal page loads and API calls pass through. If something legitimate is blocked, disable that rule and check the Block log for the exact path and pattern. Adjust CDN/WAF rules in production the same way.

        Do I still need to fix Code findings if Protect blocks them?

        Yes. WAF blocks are a bandage — fix the underlying bug in source, tighten IaC, and add headers. Protect catches automated spray-and-pray attacks; it does not patch your codebase.

        Why sync if built-in rules already exist?

        Built-ins cover generic attacks. Sync adds context-aware rules from your findings — e.g. if Cloud flagged open security groups, we also block metadata SSRF probes relevant to that posture.

        How do I deploy Protect in production?

        Export the pattern column from the rules table and implement equivalent rules in Cloudflare WAF, AWS WAF, ModSecurity, or your framework middleware. This dashboard proves the patterns work before you roll them out.

        What happens when a request is blocked?

        The client receives HTTP 403 with a short message. The event is logged in Block log with the rule ID — use it to tune false positives or prove the WAF is working in audits.
